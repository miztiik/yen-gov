# Collection inventory

**Last Updated**: 2026-05-16
**Status**: canonical (since `indicator.schema.json` v4.0, [ADR-0026](../architecture/decisions/0026-lift-collection-inventory-out-of-indicator-artifact.md))

Collection inventory answers "where do we stand on collecting this
series?". Since v4.0 it no longer lives **inside** the indicator
artifact — it is split across two sibling files in
`datasets/reference/in/`, one fully derived and one hand-edited.
The folded indicator itself (see [folded-indicator](folded-indicator.md))
carries only `series_spec.description`, `methodology`, `rows[]`,
`coverage`, and `sources[]`.

## The two files

| File | Authoring | Schema | Role |
| --- | --- | --- | --- |
| [`indicators-completeness.json`](../../datasets/reference/in/indicators-completeness.json) | **Derived** — regenerated by `tools/emit_indicators_completeness_index.py` on every ingest. Never hand-edited. | [`indicators-completeness.schema.json`](../../datasets/schemas/indicators-completeness.schema.json) | Per-indicator `{status, observed_periods, observed_geographies, last_collected_at, ...}`. The citizen `/data-completeness` page and admin Indicators panel both read this. |
| [`indicators-operator-state.json`](../../datasets/reference/in/indicators-operator-state.json) | **Hand-edited** (sparse — most indicators have no entry). | [`indicators-operator-state.schema.json`](../../datasets/schemas/indicators-operator-state.schema.json) | Per-indicator overlay of `{frozen, refetch_requested, unavailable_periods}`. The ONE writable knob for operators. |

The derived file ALSO consumes the operator-state overlay so its
`unavailable_periods` reflect operator triage; that is the only
place the two files cross.

## Field ownership

| Field                 | Lives in                         | Written by                              | Read by                                                  |
| --------------------- | -------------------------------- | --------------------------------------- | -------------------------------------------------------- |
| `status`              | `indicators-completeness.json`   | Derived on emit                         | Citizen UI, `/data-completeness` index, admin Indicators |
| `observed_periods`    | `indicators-completeness.json`   | Derived from `rows[].time`              | Citizen UI                                               |
| `observed_geographies`| `indicators-completeness.json`   | Derived from `rows[].entity_id`         | Citizen UI                                               |
| `last_collected_at`   | `indicators-completeness.json`   | Derived: `max(sources[].fetched_at)`    | Citizen UI; planner ignores                              |
| `frozen`              | `indicators-operator-state.json` | **Operator** (admin Indicators panel)   | Planner (skip whole indicator), citizen UI               |
| `refetch_requested`   | `indicators-operator-state.json` | **Operator** triage flag                | Admin Indicators; planner clears after re-collect        |
| `unavailable_periods` | `indicators-operator-state.json` | **Adapter or operator** structured exclusion | Completeness emitter, citizen UI                    |

Derivation source: [`backend/yen_gov/inventory/derive.py`](../../backend/yen_gov/inventory/derive.py)
is preserved for the historical v1.5→v2.0 migration tool; the
daily-driver emitter is
[`tools/emit_indicators_completeness_index.py`](../../tools/emit_indicators_completeness_index.py).

## Why two files, not one block

See [ADR-0026](../architecture/decisions/0026-lift-collection-inventory-out-of-indicator-artifact.md)
for the full rationale. Briefly: keeping a derived block AND operator
flags AND publisher-promise fields all inside the artifact merged three
incompatible lifecycles (per-fetch / per-collect-run / hand-authored)
and kept regenerating the artifact every time the inventory churned —
the same `fetched_at`-smear shape the project already fought elsewhere.
The lift makes each lifecycle visible in its own file.

## Period tokens — `{key, label, frequency}`

The operator-state overlay's `unavailable_periods[].period` carries
the `{key, label, frequency}` shape:

- `key` — stable equality token. Normally equals `rows[].time`. Not
  citizen-facing. Used for set membership only.
- `label` — the **publisher's exact period string**: `"FY 2024-25"`,
  `"as on 31.03.2025"`, `"Census 2011"`, `"Q3 2024-25"`. MUST be
  citizen-readable without unwrapping. The adapter (or operator) is
  the single authority on these labels.
- `frequency` — one of nine enum values:
  `annual_fy`, `annual_cy`, `quarterly_fy`, `quarterly_cy`, `monthly`,
  `weekly`, `daily`, `decennial`, `ad_hoc`.

Nothing parses these labels. There is no normaliser, no LLM, no
canonical-form transformer anywhere in this path. The Indian
publisher vocabulary is irreducibly diverse; any attempt to flatten
it loses information and creates a new round-tripping class of bugs.

## `expected_periods` was removed in v4.0

v3.0 carried `series_spec.expected_geographies` and `expected_periods`
(plus an `expected_periods_inference` enum) as a publisher-promise
declaration. v4.0 removed all three. The framework no longer pretends
to know a publisher-promised cell universe at the schema level. If a
particular indicator's publisher publishes a release calendar, the
adapter authors that into `methodology.notes` as plain prose where it
makes sense; the citizen-facing completeness story is told entirely
from **what we have observed** plus the operator's **structured
unavailability claims**, never from a speculative expected universe.

See [ADR-0026](../architecture/decisions/0026-lift-collection-inventory-out-of-indicator-artifact.md)
for why the publisher-promise tier was the wrong altitude.

## Adapter writes citizen-readable labels

Documentation rule, not validator rule: the adapter — the only code
that knows what the publisher actually printed — MUST write labels in
the publisher's own form. If the adapter writes `"2024-Q3"` when the
publisher printed `"Q3 2024-25"`, then citizens see the adapter's
abbreviation and the round-trip back to upstream breaks. The schema
cannot enforce this; review must.

## `rm` is the only force-recollect

There is no force-refetch flag. To re-collect an indicator from
scratch, the operator deletes the relevant `.runtime/raw/` files (per
[ADR-0003](../architecture/decisions/0003-no-fetch-cache.md)) and
re-runs the collector. See [How-to: force re-collection](../how-to/force-recollect.md).

`refetch_requested` is **triage status**, not a second force mechanism.
An operator marks an indicator `refetch_requested: true` to flag "this
needs a re-pull on the next operator pass"; the planner clears the
flag to `false` after a successful re-collect, so the absence of a
flag is also signal.
